## 项目与边界

本目录用于 `tinyexpr-rs/` 翻译实验。  
`../tinyexpr-c/` 仅作参考，禁止修改。  
目标是通过 `../scripts/compare.ps1` 与 C 基准对齐。

## 交互协议（必须遵守）

1. 先解释再实现：Python 类比 → 翻译决策 → 风险点 → 代码。
2. 小步迭代：每次仅处理一个函数或一组紧耦合定义。
3. 即时验证：每次改动后给出最小测试与验证建议。
4. 日志由用户编写：可提醒记录点，不代写日志或 findings。

## 执行自主性（减少频繁提问）

- 默认采用自主闭环：实现 -> 本地测试 -> 修复 -> 再测试 -> 汇报结果。
- 在同一子任务内，不要为每个小步骤请求确认；优先连续执行到可验证状态。
- 仅在以下关键情形提问：架构级取舍、需求歧义、潜在破坏性改动、连续失败或外部阻塞。
- 汇报以阶段性结果为主，不用高频中断式确认。

## 日志强约束（raw journal）

- `../logs/raw-journal.md` 是本实验的唯一原始过程记忆，必须持续追加。
- 关键发现要提醒用户记录。
- 出现以下事件时必须提醒记录：关键决策、测试失败、语义差异、修复完成、阻塞点。
- 只给记录要点，不代写日志正文；禁止跳过日志提醒继续实现。

## 技术约束

- 工具链：`stable-x86_64-pc-windows-gnu`
- 非必要不引入新依赖
- 非必要不使用 `unsafe`，如使用必须说明边界与原因
- 默认采用 Rust 惯用法，不机械直译

## 错误处理策略

- 先诊断再改码：先快速定位最可能原因，再直接实施验证与修复闭环。
- 明确不确定性：无法确认时必须显式说明。
